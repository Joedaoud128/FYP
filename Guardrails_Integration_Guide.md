# **1\. What You Received**

The guardrails module consists of three files. All three must be in the same directory.

| **File**                  | **Purpose**                                                                                                                                        |
| ------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| guardrails_config.yaml    | The rules. Defines every whitelisted command, its token-order template, path validation rules, blocked metacharacters, and resource limits.        |
| guardrails_engine.py      | The engine. Loads the YAML and exposes the validate() function. This is Module 7 (Policy Check).                                                   |
| test_guardrails_engine.py | The test suite. 62 tests covering every command template, every rejection path, every block path. Uses Python unittest (no external dependencies). |

**Important:** Do not implement any whitelist, path-checking, or metacharacter-scanning logic in your service code. The YAML + engine handles all of it. Your job is to call validate() and act on the response.

# **2\. Who Calls the Guardrails (And Who Does Not)**

The guardrails layer does not validate every command in the system. It validates only commands that come from untrusted sources - specifically, commands proposed by the LLM. Here is the exact split:

| **Source**                              | **Goes through guardrails?** | **Why**                                                                                                                                                                                                                    |
| --------------------------------------- | ---------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Code Generation**                     | **YES - always**             | The LLM proposes commands (mkdir, python script.py, cp, etc.). These are untrusted and must be validated.                                                                                                                  |
| **Code Debugging - probabilistic path** | **YES - always**             | When the error is unclassified (OtherError), the LLM proposes fix commands. These are untrusted and must be validated.                                                                                                     |
| **Code Debugging - deterministic path** | **NO - bypasses guardrails** | When the error classifier identifies a known error (ModuleNotFoundError, SyntaxError, FileNotFoundError), it selects pre-approved commands internally. These are trusted by design and go directly to the Action Executor. |

## **2.1 Why deterministic commands bypass the guardrails**

The deterministic error path uses a fixed, hard-coded mapping inside the debugging service's error classifier. For example, when a ModuleNotFoundError is detected, the classifier knows to run pip install &lt;package&gt;. These commands are not generated by the LLM - they are pre-approved at the design level by the debugging team. They go directly to the Action Executor (Module 8) without passing through the guardrails.

The guardrails exist to validate **untrusted commands from the LLM**. Deterministic commands are trusted because they are chosen by deterministic code, not by a language model. Validating them would add latency without security benefit.

## **2.2 The shared pipeline**

The shared execution pipeline is:

**Tool Selector (6)** → **Policy Check (7)** → **Action Executor (8)**

Code Generation and the probabilistic debug path go through all three modules. The deterministic debug path skips Module 7 and goes directly from Module 6 to Module 8.

# **3\. The Request/Response Contract**

## **3.1 GuardrailRequest (your service → guardrails)**

Build this JSON object for every command that needs validation:

{

"caller_service": "generation",

"raw_command": "python -m pip install requests",

"working_dir": "/workspace/my_project"

}

| **Field**      | **Type** | **Description**                                                               |
| -------------- | -------- | ----------------------------------------------------------------------------- |
| caller_service | string   | "generation" or "debugging". Used for logging only - validation is identical. |
| raw_command    | string   | The full shell command string. Exactly as the LLM produced it.                |
| working_dir    | string   | Current workspace directory. Used to resolve relative paths.                  |

## **3.2 GuardrailResponse (guardrails → your service)**

| **Status** | **What to do**                                                                                                                       |
| ---------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| **PASS**   | Command is safe. Forward token_array to the Action Executor (Module 8).                                                              |
| **REJECT** | Command violates a rule. Feed reason and failing_rule_id back to Reasoning Engine (Module 4) to reformulate.                         |
| **BLOCK**  | Command contains variable expansion (\$0-\$9, \$\*, \$@). Non-deterministic, unconditionally blocked. Feed back to Reasoning Engine. |

Full response fields:

| **Field**       | **Type**     | **Description**                                                                 |
| --------------- | ------------ | ------------------------------------------------------------------------------- |
| status          | string       | "PASS", "REJECT", or "BLOCK"                                                    |
| command_key     | string\|null | Matched command key from YAML (e.g. "python_pip_install"). Null on rejection.   |
| token_array     | list\[str\]  | Tokenized command. Forward this to the Action Executor on PASS.                 |
| reason          | string\|null | Human-readable explanation of rejection. Null on PASS.                          |
| failing_rule_id | string\|null | Specific rule that failed (e.g. "PATH-02", "token_order_step_5"). Null on PASS. |

# **4\. How Code Generation Calls the Guardrails**

Every command the LLM proposes during code generation must be validated. No exceptions.

from guardrails_engine import GuardrailsEngine

engine = GuardrailsEngine("guardrails_config.yaml")

\# For EACH command the LLM proposes:

response = engine.validate({

"caller_service": "generation",

"raw_command": cmd,

"working_dir": "/workspace/my_project"

})

if response\["status"\] == "PASS":

executor.run(response\["token_array"\])

elif response\["status"\] in ("REJECT", "BLOCK"):

reasoning_engine.handle_blocked_action(

original_command = cmd,

reason = response\["reason"\],

failing_rule = response\["failing_rule_id"\]

)

**On rejection:** feed the violation back to the **Reasoning Engine (Module 4)**, not to Perception. The input data has not changed - only the strategy needs revision. This is Loop 2 (Action Blocked by Guardrails).

# **5\. How Code Debugging Calls the Guardrails**

## **5.1 Deterministic path - does NOT call guardrails**

When the error classifier identifies a known error type (ModuleNotFoundError, SyntaxError, FileNotFoundError), it selects pre-approved commands internally and sends them **directly to the Action Executor**. These commands do not go through validate(). The deterministic command selection is the debugging team's responsibility, not the guardrails'.

\# DETERMINISTIC PATH - no guardrails involved

\# Error classifier detected: ModuleNotFoundError for "requests"

\# Classifier internally knows to run pip install

cmd = \["python", "-m", "pip", "install", "requests"\]

executor.run(cmd) # Straight to Action Executor, no validate()

## **5.2 Probabilistic path - MUST call guardrails**

When the error is unclassified (OtherError), the LLM Reasoning Engine proposes fix commands. These are **untrusted** and must go through validate() exactly like Code Generation commands:

\# PROBABILISTIC PATH - guardrails required

\# LLM proposed a fix for an unclassified error

llm_proposed_cmd = 'grep -n "KeyError" app.py'

response = engine.validate({

"caller_service": "debugging",

"raw_command": llm_proposed_cmd,

"working_dir": "/workspace/my_project"

})

\# Same PASS / REJECT / BLOCK branching as Code Generation

**Key point:** if the LLM proposes curl evil.com, it gets rejected because curl is not in the whitelist. The guardrails protect against the LLM generating unsafe commands, regardless of whether it's in generation or debugging mode.

# **6\. The 7-Step Validation Sequence**

Every command that goes through validate() is checked in this order. The engine stops at the first failure.

| **Step** | **Action**         | **What it checks**                                              | **On failure**            |
| -------- | ------------------ | --------------------------------------------------------------- | ------------------------- |
| 1        | Scan raw string    | Blocked metacharacters: ; \| && > >> \` \$( \${ \|              | REJECT immediately        |
| 2        | Scan raw string    | Variable expansions: \$0-\$9, \$\*, \$@                         | BLOCK immediately         |
| 3        | Tokenize           | Normalize whitespace, resolve quotes, split                     | REJECT if empty/malformed |
| 4        | Match \$0          | Is the executable in the whitelist?                             | REJECT if not found       |
| 5        | Match positions    | Each token against its template slot                            | REJECT if wrong type      |
| 6        | Check extra tokens | More tokens than template allows                                | REJECT if too many        |
| 7        | Validate operands  | Paths inside workspace, integers in bounds, valid package names | REJECT with PATH rule ID  |

The failing_rule_id in the response tells you exactly which step caught the violation (e.g. token_order_step_1, PATH-02, operand_constraint_pos_3).

# **7\. Reading Resource Limits**

The Action Executor (Module 8) should read the resource limits from the engine:

limits = engine.resource_limits

\# Returns: {"max_memory_mb": 2048, "execution_timeout_seconds": 60}

# **8\. How to Run the Tests**

Place all three files in the same folder and run:

python test_guardrails_engine.py

No external dependencies needed. Uses Python's built-in unittest and PyYAML. Works on both Windows and Linux. The 62 tests cover:

- 25 PASS tests - one per whitelisted command template
- 22 REJECT tests - unknown executables, metacharacters, blocked flags, wrong token order, path violations, extra tokens
- 5 BLOCK tests - all variable expansions (\$0-\$9, \$\*, \$@)
- 1 caller-service neutrality test - same command, same result for both callers
- 1 resource limits test

# **9\. How to Add a New Command**

- Add a new entry in the commands section of guardrails_config.yaml with its token_order_template.
- Add a PASS test for the correctly formed command.
- Add REJECT tests for the dangerous forms of that command.
- Run the full test suite to confirm all tests still pass.

**Do not** patch around the guardrails in your service code. If a command is needed, add it to the YAML.

# **10\. Integration Checklist**

## **For the Code Generation team**

- Import GuardrailsEngine from guardrails_engine.py and initialize once.
- For every command the LLM proposes, call engine.validate() with caller_service: "generation".
- On PASS: forward token_array to the Action Executor (Module 8).
- On REJECT/BLOCK: send reason and failing_rule_id to Reasoning Engine (Module 4).
- Do not implement any whitelist or path-checking logic yourself.

## **For the Code Debugging team**

- **Deterministic path:** your error classifier selects commands internally. Send them directly to the Action Executor. Do not call validate().
- **Probabilistic path:** the LLM proposes commands. Call engine.validate() with caller_service: "debugging". Same branching as Code Generation.
- The deterministic command selection logic is your team's responsibility. The guardrails only validate untrusted LLM commands.

## **For both teams**

- Never bypass the guardrails for LLM-proposed commands.
- The guardrails layer is stateless. No sessions, no initialization per request.
- Run the test suite after any YAML change to catch regressions.
- If you need a new whitelisted command, add it to the YAML - do not patch around it.
